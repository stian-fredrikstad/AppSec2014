XML Signature

HMAC Truncation Attack

		<SignatureMethod Algorithm="http://www.w3.org/2000/09/xmldsig#hmac-sha1">
			<HMACOutputLength>1</HMACOutputLength>
		</SignatureMethod>

Settes dette parameteret til 1 vil det være 50% sannsynlig at signaturverdien godtas i litt for mange rammeverk

Security of XML-based Web Services and Single Sign-On

XML Signature Wrapping

  • Legg til en ekstra body med samme id som den signerte
  • Når meldingen skal verifiseres, så verifiseres den ekte
  • Når applikasjonen skal bruke meldingen, så brukes den nye bodyen
  • Kan f.eks. utføres på tjenester med SAML, og den mest kjente norske er jo bankID
  • Gitt at service provide (les: banken) ikke har en riktig implementasjon, så kan vi catche meldingen før SAML response sendes til service provider og f.eks. modifisere personnr
  • Dette er jo kanskje et problem som bør sees nærmere på, selv om bankID har masse "kryptogreier", så er det fortsatt opp til service provider å godkjenne/tolke SAML responsen

Security of XML-based Web Services and Single Sign-On

XML Signature Exclusion

  • Enkelt og greit fjern hele signaturelementet
  • Skjedde f.eks. mot Amazon EC2
  • Fungerte mot:
  • • Apache Axis2
  • • JOSSO
  • • OpenAthens

Security of XML-based Web Services and Single Sign-On

XML SAML Certificate Confusion

  • En SAML response kan inneholde et sertifikat som brukes til meldingen
  • Dette kan være self signed, som er greit nok, men service provider må sjekke om dette er et sertifikat som godkjennes

Security of XML-based Web Services and Single Sign-On

XML SAML Provider Confusion

  • En SAML response har en destination som den er ment for
  • Denne kan f.eks. være http://banken.no eller http://evil.com
  • Det er ingenting i veien for at http://evil.com bruker samme identity provider som http://banken.no, men identity provider vil sette en destination som meldingen er ment for (og som forhåpentligvis er signert), da er det viktig at http://banken.no ikke godkjenner andre meldinger enn de som er ment for den
  • http://evil.com kan i motsatt fall lure en bruker til å logge inn på vegne av http://evil.com og deretter bruke denne meldingen serverside for å få tilgang til http://banken.no
